🛠️ Phase 1: Data & Code Versioning

Objective: Create a reproducible environment where code and heavy data are versioned separately but stay in sync.
Git DVC GitHub Local Remote (simulates S3/GCS)
💡 Why DVC? The Problem It Solves

Git is excellent for code (small text files) but terrible for large data files — a 5GB CSV would bloat your repository and slow every clone. DVC solves this with a "pointer pattern":

Git tracks iris.csv.dvc
~100 bytes · the pointer
+
DVC stores iris.csv
actual data → remote
=
Full reproducibility
code + data in sync
FeatureGit OnlyGit + DVC
Code versioning✅ Yes✅ Yes
Large files (GB+)❌ Breaks repo✅ Stored in remote
Data-code sync❌ Manual✅ Automatic via .dvc files
Reproduce past experiment⚠️ Hard✅ git checkout + dvc pull
1️⃣ Repository Setup
Initialize local Git repo and link to GitHub

First create your project folder, initialize git, and connect it to a remote GitHub repository.

bash
# 1. Create project structure mkdir mlops-project && cd mlops-project mkdir -p data models tests reports .github/workflows # 2. Initialize Git git init Initialized empty Git repository in /home/user/mlops-project/.git/ # 3. Create .gitignore (never commit raw data or model binaries) cat > .gitignore << 'EOF' # Data files — tracked by DVC, not Git data/*.csv data/*.parquet data/*.json # Model binaries models/*.pkl models/*.joblib models/*.h5 # MLflow runs mlruns/ mlartifacts/ # Python artifacts __pycache__/ *.pyc .env venv/ EOF # 4. Link to GitHub remote git remote add origin https://github.com/YOUR_USERNAME/mlops-project.git # 5. First commit git add .gitignore git commit -m "feat: initial project setup with .gitignore" [main (root-commit) a1b2c3d] feat: initial project setup with .gitignore 1 file changed, 18 insertions(+) git push -u origin main Branch 'main' set up to track remote branch 'main' from 'origin'.
✅ Result: You now have a Git repo linked to GitHub. The .gitignore ensures large data files and model binaries are never accidentally committed to Git.
2️⃣ DVC Initialization & Local Remote
Install DVC and configure a local remote (simulates S3/GCS)

A "remote" in DVC is where the actual data files are stored. In production this would be S3, GCS, or Azure Blob. Here we use a local folder to simulate that.

bash
# Install DVC pip install dvc Successfully installed dvc-3.x.x # Initialize DVC inside the Git repo dvc init Initialized DVC repository. You can now commit the changes to git. +---------------------------------------------------------------------+ | DVC has enabled anonymous aggregate usage analytics. Read the | | analytics documentation (and how to opt-out) here: | | https://dvc.org/doc/user-guide/analytics | +---------------------------------------------------------------------+ # dvc init creates .dvc/ folder and .dvcignore — commit them git add .dvc .dvcignore git commit -m "feat: initialize DVC" # Create the "remote" folder (outside your project = separate storage) mkdir -p ~/mlops-dvc-remote # Register it as DVC remote called 'local_remote' dvc remote add -d local_remote ~/mlops-dvc-remote Setting 'local_remote' as a default remote. # Commit the updated .dvc/config git add .dvc/config git commit -m "config: add DVC local remote storage"
📄 .dvc/config (auto-generated)TOML
[core] remote = local_remote autostage = true # auto-add .dvc files to git ['remote "local_remote"'] url = /home/user/mlops-dvc-remote # → swap for s3://bucket/path in prod
🔧 Production Swap: To use real cloud storage, replace the url with:
dvc remote add -d myremote s3://my-bucket/dvc-data
dvc remote add -d myremote gs://my-bucket/dvc-data
DVC handles authentication via dvc remote modify or environment variables.
3️⃣ Data Tracking with DVC
Add iris.csv to DVC tracking

dvc add moves the file under DVC control, creating a tiny pointer file (.dvc) and updating .gitignore to exclude the raw data from Git.

bash
# Add dataset (150 rows, 6 cols) dvc add data/iris.csv To track the changes with git, run: git add data/iris.csv.dvc data/.gitignore To enable auto staging, run: dvc config core.autostage true # Examine what was created cat data/iris.csv.dvc outs: - md5: 4c4017d7c5357576db6a89f8cac99631 # content hash (fingerprint) size: 4185 # bytes path: iris.csv # Check .gitignore was updated automatically cat data/.gitignore /iris.csv # DVC added this — iris.csv is now ignored by Git # Verify structure ls -la data/ -rw-r--r-- 1 user 4185 Mar 10 iris.csv # actual data (tracked by DVC) -rw-r--r-- 1 user 95 Mar 10 iris.csv.dvc # pointer file (tracked by Git) -rw-r--r-- 1 user 11 Mar 10 .gitignore # updated by DVC
🔑 The MD5 Hash is the Key: The hash 4c4017d7... is a fingerprint of your data's exact content. Git commits the .dvc pointer. DVC uses the hash to find the correct data in remote storage. If you modify iris.csv, the hash changes → DVC detects it → you create a new versioned snapshot.
4️⃣ The Git + DVC Workflow
Push data to DVC remote, commit metadata to Git

The golden rule: Git commits the small .dvc files. DVC pushes the heavy data files. They stay linked via the MD5 hash.

bash
─── PUSH ────────────────────────────────────────────────────────── # Step 1: Push heavy data to DVC remote (simulates S3 upload) dvc push 1 file pushed # iris.csv → ~/mlops-dvc-remote/4c/4017d7... # Step 2: Commit the lightweight metadata to Git git add data/iris.csv.dvc data/.gitignore git commit -m "data: add iris.csv (v1, 150 rows) via DVC" [main 2f3e4d5] data: add iris.csv (v1, 150 rows) via DVC 2 files changed, 6 insertions(+) git push origin main To https://github.com/YOU/mlops-project.git a1b2c3d..2f3e4d5 main → main ─── PULL (team member or CI server) ─────────────────────────────── # Clone code (fast — only .dvc files, no actual data) git clone https://github.com/YOU/mlops-project.git # Download the actual data from DVC remote dvc pull Collecting |1.00 [00:00, 2.33entry/s] Fetching |1.00 [00:00, 15.3file/s] 1 file added ─── UPDATE DATA (new version) ───────────────────────────────────── # Replace with v2 data (new MD5 hash auto-detected) cp iris_v2.csv data/iris.csv dvc add data/iris.csv To track the changes with git, run: git add data/iris.csv.dvc dvc push && git add data/iris.csv.dvc && git commit -m "data: update iris.csv to v2" # Reproduce old experiment with v1 data any time: git checkout HEAD~1 -- data/iris.csv.dvc && dvc pull
✅ Phase 1 Complete! Your project now has full data + code versioning. Anyone on your team can clone the repo, run dvc pull, and reproduce the exact experiment environment.

1️⃣ MLflow Integration with autolog()
One line to log everything: mlflow.sklearn.autolog()

autolog() automatically captures: model parameters, training metrics, feature importances, and the model artifact itself — no manual logging needed.

📄 train.py — Core StructurePython
import mlflow import mlflow.sklearn from sklearn.ensemble import RandomForestClassifier from sklearn.metrics import accuracy_score, f1_score # ── Set the experiment name (groups all runs together) ────────────── mlflow.set_experiment("iris-classification") # ── ONE LINE: captures params, metrics, model automatically ───────── mlflow.sklearn.autolog() with mlflow.start_run(run_name=f"RF_n{n_estimators}_d{max_depth}"): # Additional manual params not captured by autolog mlflow.log_param("learning_rate", learning_rate) # Train model = RandomForestClassifier( n_estimators=n_estimators, max_depth=max_depth, random_state=42 ) model.fit(X_train, y_train) # autolog captures all RF params here preds = model.predict(X_test) # Log extra metrics mlflow.log_metrics({ "accuracy": accuracy_score(y_test, preds), "f1_score": f1_score(y_test, preds, average="weighted"), }) # autolog also saves the model as an artifact run_id = mlflow.active_run().info.run_id print(f"Run ID: {run_id}")

1️⃣ FastAPI Development
Build a production-grade REST API with Pydantic validation

Pydantic automatically validates incoming JSON, returning clear error messages if fields are missing or wrong types — before your model even sees the data.

📄 main.py — Complete APIPython
import pickle import numpy as np from fastapi import FastAPI, HTTPException from pydantic import BaseModel, Field # ── Pydantic schema: defines and VALIDATES the input ────────────────── class IrisFeatures(BaseModel): sepal_length: float = Field(..., gt=0, example=5.1) sepal_width: float = Field(..., gt=0, example=3.5) petal_length: float = Field(..., gt=0, example=1.4) petal_width: float = Field(..., gt=0, example=0.2) class PredictionResponse(BaseModel): species: str class_id: int confidence: float probabilities: dict # ── App setup ────────────────────────────────────────────────────────── app = FastAPI(title="Iris Classifier API", version="1.0.0") SPECIES = {0: "setosa", 1: "versicolor", 2: "virginica"} # Load once at startup (not per-request = fast) with open("models/rf_model.pkl", "rb") as f: model = pickle.load(f) @app.get("/") def root(): return {"status": "ok", "message": "Iris Classifier API 🌸"} @app.post("/predict", response_model=PredictionResponse) def predict(features: IrisFeatures): X = np.array([[features.sepal_length, features.sepal_width, features.petal_length, features.petal_width]]) class_id = int(model.predict(X)[0]) proba = model.predict_proba(X)[0] confidence = float(proba[class_id]) return PredictionResponse( species=SPECIES[class_id], class_id=class_id, confidence=round(confidence, 4), probabilities={SPECIES[i]: round(float(p), 4) for i, p in enumerate(proba)}, )
🛡️ Why Pydantic? Without Pydantic, if someone sends "sepal_length": "hello", your model crashes with an unhandled exception. With Pydantic, FastAPI automatically returns: 422 Unprocessable Entity: value is not a valid float — clean, safe, and automatic.

2️⃣ Data Drift Detection with Evidently AI
Compare training distribution vs live serving data

Data drift occurs when the statistical properties of your live data change compared to what the model was trained on. This silently degrades model accuracy without any error.

📄 drift_report.py — Core LogicPython
# Option A: Evidently AI (rich reports, recommended for production) from evidently.report import Report from evidently.metric_preset import DataDriftPreset report = Report(metrics=[DataDriftPreset()]) report.run(reference_data=train_df, current_data=serve_df) report.save_html("reports/drift_report.html") # Option B: Manual KS Test (no extra dependencies) from scipy import stats for feature in FEATURES: ks_stat, p_val = stats.ks_2samp( train_df[feature], serve_df[feature] ) drifted = p_val < 0.05 # 95% confidence threshold print(f"{feature}: KS={ks_stat:.4f}, p={p_val:.4f}, drift={drifted}")

📖 How to Interpret: The KS Statistic measures the maximum vertical distance between two cumulative distribution functions. A value of 0 = identical distributions. A value of 1 = completely non-overlapping. When p-value < 0.05, we reject the null hypothesis that both samples come from the same distribution — meaning the feature has statistically significant drift.
🚨 Action Required: 3/4 features show drift (p < 0.05). Recommended response:
1. Collect new labeled data from the current distribution.
2. Retrain the model on the updated dataset.
3. Re-validate against the performance gate (accuracy ≥ 0.90).
4. Promote new model version through staging → production in MLflow Registry.
